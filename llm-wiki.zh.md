# LLM Wiki（中文整理版）

一种用 LLM 构建个人知识库的模式。

> 这是一个“思路文档”，适合复制给你的 LLM 代理（如 Claude Code、Codex 等），由它和你一起落地具体实现。

## 1. 核心思想

多数人用 LLM + 文档的方式是 RAG：每次提问时从原始文档里临时检索、临时拼装答案。能用，但不积累。

LLM Wiki 的思路不同：

- 不只在查询时“取片段”，而是让 LLM **持续维护一个持久化 wiki**（结构化、互相链接的 Markdown 页面集合）
- 新资料进入后，LLM 会：提炼关键信息、更新实体页/主题页、标注新旧观点冲突、更新综合结论
- 知识不是每次从零推导，而是“编译一次、持续增量更新”

**关键差异：wiki 是可复利的长期资产。**

- 交叉引用提前建立
- 矛盾提前标注
- 综合结论持续演进

你通常不手写 wiki；你负责选资料和提问题，LLM 负责总结、归档、关联与维护。

一个直观比喻：

- Obsidian 像 IDE
- LLM 像程序员
- Wiki 像代码库

## 2. 适用场景

- **个人成长**：目标、健康、心理、日志与播客笔记沉淀
- **研究工作**：论文/报告长期跟踪，形成动态论证
- **读书伴随**：章节、角色、主题、线索联动（类似个人版 fan wiki）
- **团队/业务**：把 Slack、会议纪要、项目文档、客户沟通沉淀为可维护内部知识库
- **其他**：竞品分析、尽调、旅行规划、课程笔记、兴趣深潜等

## 3. 三层架构

### 3.1 Raw sources（原始资料层）

- 你精选的原始输入：文章、论文、图片、数据文件
- 只读不可改，作为事实源（source of truth）

### 3.2 Wiki（知识页层）

- LLM 生成和维护的 Markdown 页面目录
- 包含：摘要页、实体页、概念页、对比页、总览页、综合分析页
- 由 LLM 负责创建、更新、互链和一致性维护

### 3.3 Schema（规则层）

- 一个规则文档（如 CLAUDE.md / AGENTS.md）
- 规定：目录结构、命名规范、摄取流程、问答流程、维护流程
- 它决定 LLM 是“纪律化维护者”而非“随聊机器人”

## 4. 核心操作流程

### 4.1 Ingest（摄取）

把新资料放进 raw 后让 LLM 处理，典型步骤：

1. 读取资料
2. 与你讨论重点
3. 写入摘要页
4. 更新索引
5. 更新相关实体/概念页
6. 追加日志

一份资料可能影响 10–15 个页面。可单条精细摄取，也可批量摄取。

### 4.2 Query（查询）

围绕 wiki 提问，LLM 读取相关页面并带引用地综合回答。

输出形式可变：

- Markdown 页面
- 对比表
- 幻灯片（Marp）
- 图表（matplotlib）
- 画布等

重要实践：**把高价值问答回写 wiki**，让探索本身也沉淀复利。

### 4.3 Lint（巡检）

定期让 LLM 做健康检查：

- 页面间矛盾
- 被新证据淘汰的旧结论
- 无入链孤儿页
- 只被提及但无独立页面的关键概念
- 缺失交叉引用
- 可通过检索补齐的数据缺口

## 5. 两个关键文件：索引与日志

### 5.1 `index.md`（内容导向）

- 全站目录：页面链接 + 一句话摘要（可带日期、来源数等元数据）
- 按类别组织（实体、概念、来源等）
- 查询时先读 index，再下钻具体页面

中等规模下（约 100 份来源、数百页面）通常足够，不一定需要向量检索基础设施。

### 5.2 `log.md`（时间导向）

- 追加式时间线：摄取、查询、巡检记录
- 建议统一前缀格式，便于命令行解析
- 用于回看演化过程与近期动作

## 6. 可选 CLI 工具

随着 wiki 增长，可加工具提升效率。

- 首选能力：站内搜索
- 小规模可仅靠 `index.md`
- 大规模可用 [qmd](https://github.com/tobi/qmd)：本地 Markdown 搜索（BM25/向量混合 + LLM 重排），支持 CLI 与 MCP
- 也可让 LLM 快速写一个简单搜索脚本

## 7. 实用技巧

- **Obsidian Web Clipper**：网页转 Markdown，快速入库
- **图片本地化**：下载附件到固定目录，避免外链失效，便于 LLM 读取图片
- **图谱视图（Graph）**：快速识别枢纽页与孤儿页
- **Marp**：从 wiki 内容直接出幻灯片
- **Dataview**：若页面有 YAML frontmatter，可做动态表格/列表
- Wiki 本质是 Markdown Git 仓库：天然有版本、分支、协作能力

## 8. 为什么这种方式有效

知识库难点通常不在“阅读/思考”，而在“维护账务”：

- 跨页更新
- 交叉引用维护
- 新旧证据冲突标注
- 全局一致性维持

人会因维护成本过高而放弃；LLM 不会疲劳，且能一次改多页，维护边际成本接近零。

人类更该做的是：

- 选择来源
- 设定分析方向
- 提出高质量问题
- 做最终判断

## 9. 背景与定位

该思路与 Vannevar Bush 在 1945 年提出的 Memex 精神相近：

- 私有、精选、可追踪关联的知识空间
- 文档之间的连接与路径本身也有价值

当年难点是“谁来维护”；现在可由 LLM 承担。

## 10. 注意事项

本文是模式说明，不是固定实现。

以下都应按你的场景自定义：

- 目录结构
- schema 规范
- 页面模板
- 工具链
- 输出格式

一切模块都可选：

- 纯文本场景可忽略图片流程
- 小型 wiki 可不建搜索引擎
- 不需要演示材料就只保留 Markdown

**正确用法**：把这份模式给 LLM 代理，再一起实例化成适配你工作流的版本。
