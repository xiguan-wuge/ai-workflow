# DevDocs 知识库记录

使用这个 skill，将项目开发过程中的踩坑问题、调试过程、实现经验整理成可复用的 Markdown 知识库条目，并写入项目根目录的 `devDocs/` 目录。

## 目标

把一次问题排查、修复过程或实现经验沉淀为项目内长期可读的知识记录，方便后续开发者和 AI 代理在遇到类似问题前先查阅，避免重复踩坑。

## 记录流程

1. 从当前对话中识别具体问题、踩坑点或经验结论。
2. 如果关键信息不足，先用简短问题补齐，不要直接编造：
   - 原始现象或问题背景是什么？
   - 直接原因和根因是什么？
   - 最终采用了什么解决方案？
   - 这个经验适用于哪些项目模块、文件、包、工具或流程？
3. 在 `devDocs/` 下创建或更新 Markdown 文件。
4. 确保 `devDocs/README.md` 存在，并把它作为知识库索引入口维护。
5. 优先一个踩坑点对应一篇聚焦的文档。
6. 文件名使用简短、可读的 kebab-case，例如：
   - `devDocs/docs-deploy-base-path.md`
   - `devDocs/pnpm-frozen-lockfile.md`
   - `devDocs/vitepress-demo-plugin.md`

## 文档模板

除非用户指定其他格式，否则使用以下结构：

```markdown
# [问题或经验标题]

## 背景

[问题发生的业务/技术背景，涉及的项目模块、命令、环境或触发条件。]

## 现象

[用户看到的错误、异常表现、失败日志、页面现象或错误结果。]

## 原因

[根因分析。区分直接原因和深层原因；如果仍不确定，明确写“待确认”。]

## 解决方案

[已验证的修复步骤、关键修改、命令或配置。]

## 经验总结

[这次踩坑带来的判断规则、排查顺序、注意事项。]

## 可复用范围

[说明这个经验适用于哪些模块、工具链、类似场景，以及不适用的边界。]

## 相关文件

- `path/to/file`
```

## devDocs/README.md 索引规则

`devDocs/README.md` 是知识库入口，必须用于整理和索引所有踩坑记录，方便人工快速查看，也方便 AI 后续优先读取索引后定位具体文档。

如果 `devDocs/README.md` 不存在，创建它，并使用以下结构：

```markdown
# DevDocs 知识库索引

## 使用说明

这里记录项目开发过程中的踩坑问题、经验案例和可复用解决方案。

## 索引

| 分类 | 文档 | 简要说明 | 关键词 |
| --- | --- | --- | --- |
| 部署 | [示例标题](./example.md) | 示例说明 | vitepress, github-pages |
```

每次新增或更新 `devDocs/*.md` 时，同步维护索引表：

- `分类`：问题所属领域，例如部署、依赖、组件、文档、CI、发布、工具链。
- `文档`：使用相对链接指向具体 Markdown 文件。
- `简要说明`：一句话说明该文档解决什么问题。
- `关键词`：列出便于搜索的技术词、错误词、模块名或命令。

不要把完整问题内容写进 README；README 只做索引和导航。

## 写作规则

- 默认使用中文编写，除非用户明确要求使用其他语言。
- 内容要事实化、可执行，避免写成流水账。
- 不要编造根因、命令、日志或文件路径。
- 如果当前对话缺少足够信息，先询问再写入。
- 如果更新已有文档，保留有价值的旧内容，在其基础上补充或修正，不要盲目覆盖。
- 相关文件使用项目相对路径，并用反引号包裹。
- 重点写“下次如何避免或快速解决”，而不是只复盘发生了什么。

## 写入 devDocs 时的规则

- 如果 `devDocs/` 不存在，创建该目录。
- 不创建无关的文档目录。
- 不更新 README 或 VitePress 导航，除非用户明确要求。
- 完成后告诉用户创建或更新的文件路径，并简短说明记录了什么。

## 质量检查

结束前确认文档回答了以下问题：

- 发生了什么？
- 为什么发生？
- 如何解决？
- 下次遇到应该怎么判断和处理？
- 这个经验可以在哪些场景复用？
